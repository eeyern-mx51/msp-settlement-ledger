Payout Progression Controls

How do we control who and what can move a payout forward? A framework for discussion.

The simple idea

Every payout step (prepare, approve, transfer) can happen manually (human clicks a button) or automatically (system does it on schedule). We need controls to turn each off — at different scopes — without making everyone's head spin.

The payout pipeline

Preparation
Manual / Auto
→
Approval
Manual / Auto
→
Transfer
Manual / Auto

Each step has two switches: one for humans, one for the system.
Each switch can be flipped at three scopes: global, merchant, or individual payout.

1 Three scopes — widest wins

Global
Entire fleet
Stop everything across all merchants
"Pause fleet payouts" — the emergency brake. Blocks all steps for every merchant.
e.g. System incident, regulatory hold, suspected platform-wide issue
Merchant
Single MID
Stop payouts for one merchant
"Pause merchant payouts" — blocks all steps for a specific MID.
e.g. Fraud investigation on one merchant, bank details changing
Payout
Single payout
Hold one specific payout
"Place on hold" — blocks the next step for this payout only. Other payouts for the same merchant keep flowing.
e.g. Amounts look wrong, need to verify with merchant before approving

Override rule

Global
broadest
Merchant
narrows
Payout
most specific

If any wider scope is paused, narrower scopes can't override it.
A merchant-level resume can't bypass a global pause. A payout-level hold release can't bypass a merchant pause.

Think of it like a circuit breaker: if the main switch is off, flipping individual room switches does nothing.

2 Manual vs Automatic — who gets stopped?

🔑 FinOps Tier 1 (senior)

  • Can toggle global pause on/off
  • Can toggle merchant pause on/off
  • Can place/release payout holds
  • Can disable/enable automation per step
  • Can still act manually even when auto is off

👁 FinOps Tier 2 (junior)

  • Cannot change global pause
  • Cannot change merchant pause
  • Cannot place/release holds
  • Cannot change automation settings
  • Can view payouts (read-only)

The key insight: "disabling manual" isn't a toggle — it's a permission. Lower-level users simply don't have the buttons. Higher-level users always can, unless a scope-level pause blocks them too.

Automation toggles are independent of pause controls. You can have automation on but the merchant paused — automation just won't run for that merchant.

4 Scenarios

Scenario Scope What happens
Platform incident — DTE files are corrupted today Global FinOps T1 flips global pause. All automation stops, all manual actions blocked. Fix the data, resume.
Merchant under fraud investigation Merchant FinOps T1 pauses that MID. Other merchants keep flowing. Investigation concludes, resume merchant.
One payout has unusual amounts — need to check Payout FinOps T1 places a hold on that payout. Rest of the merchant's payouts keep flowing. Verified? Release hold.
Moving to auto-approval but want to keep manual transfers for now Global Turn on auto-approval. Leave auto-transfer off. Preparation is already manual (POC). Gradual rollout.
New FinOps T2 joins — shouldn't approve payouts Permission Not a toggle — T2 role simply doesn't have approve/execute buttons. No configuration needed.
Bank holiday — don't want transfers going out Global FinOps T1 turns off auto-transfer for the day. Payouts can still be prepared and approved, just won't execute.

Open questions for discussion

  1. Payout-level hold: Is this needed for POC, or can we ship with global + merchant scope only?
  2. Automation granularity: Do we need per-step auto toggles (prep/approve/transfer separately), or is a single "enable automation" toggle enough for now?
  3. Merchant-level auto overrides: Should a specific merchant be able to have different automation settings than the fleet default? (e.g. auto-approve for trusted merchants only)
  4. Pause reason: Should placing a pause/hold require a mandatory reason? (Useful for audit trail)
  5. Terminology: Pause / Hold / Disable progression — which term resonates best with the team?
