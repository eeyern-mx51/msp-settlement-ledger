Payout Progression Controls

Implementation guide. Updated 25 Mar 2026.

Three-pillar control model

Payout progression is controlled by three independent systems: Holds (block any progression), Automation Config (enable/disable auto-progression), and Permissions (who can act). Holds always override automation.

Holds block progression; Automation drives it forward

Preparation Hold
Blocks prep + auto-prep
→
Progression Hold
Blocks approval + transfer + auto

Two hold types across three scopes (fleet, merchant, payout).
Automation Config is separate: enables auto-progression when no holds block it.
Holds always win.

1 Holds — three scopes, two types, widest wins

Fleet
Entire fleet
Block all payout progression fleet-wide
Preparation Progression — blocks both prep and progression across all merchants.
e.g. System incident, regulatory freeze, platform-wide data issue
Merchant
Single MID
Block payout progression for one merchant
Preparation Progression — blocks prep or progression for specific MID only.
e.g. Fraud investigation, KYC pending, bank details verification
Payout
Single payout
Block one specific payout from progressing
Preparation Progression — blocks prep or progression for this payout only. Siblings keep flowing.
e.g. Amount anomaly, verification needed, settlement delay

Hold cascade rule

Fleet
broadest
Merchant
narrows
Payout
most specific

If any wider scope has a hold, narrower scopes cannot override it.
Merchant-level hold cannot bypass fleet hold. Payout-level release cannot bypass merchant hold.

Holds are hierarchical: fleet-level holds the entire pipeline; merchant-level holds one MID; payout-level holds one tx.

2 Permissions & Automation Config — independent of holds

🔑 FinOps Tier 1 (senior)

  • Can place/release holds (fleet, merchant, payout)
  • Can toggle automation per-phase (fleet, merchant)
  • Can manually trigger progression (prep, approve, execute)
  • Can view hold reasons and audit trail
  • Can resubmit failed payouts (if no progression hold)

👁 FinOps Tier 2 (junior)

  • Cannot place/release holds
  • Cannot change automation config
  • Cannot manually trigger progression
  • Can view payouts and holds (read-only)
  • Can view hold banners and reasons

Holds and Automation Config are separate. A hold blocks everything regardless of automation state. Automation Config only applies when no holds are active.

3 Automation Config — per-phase toggles

Phase Auto on Auto off Who can toggle
Preparation System auto-creates payouts daily at 8:30 AM from unsettled MLEs FinOps manually triggers "Prepare Payout" dialog FinOps T1 only
Progression
(Approval + Begin Transfer)		 System auto-approves (if rules pass) and initiates transfer FinOps manually approves and executes via UI FinOps T1 only

Holds always override automation. If a Progression hold is active, manual progression is also blocked and resubmit is disabled. Automation Config shows hold warnings in the panel.

4 Scenarios

Scenario Tool What happens
Platform incident — data corruption Fleet Hold FinOps T1 places fleet-level Preparation + Progression holds. All prep and progression blocked. Auto is irrelevant. Fix data, release holds.
Merchant under fraud investigation Merchant Hold FinOps T1 places merchant-level hold (prep or progression). That MID is frozen. Other merchants keep flowing. Investigation done? Release hold.
One payout has unusual amounts — need to verify Payout Hold FinOps T1 places payout-level Progression hold. That payout cannot advance. Siblings for same merchant flow. Amount verified? Release hold.
Failed payout needs resubmit but is stuck in progression Payout Hold Progression hold blocks resubmit. FinOps T1 must release the hold first, then resubmit is available.
Gradually rolling out auto-progression Automation Config FinOps T1 turns on Progression auto in gear panel. Payouts with no holds auto-progress. Preparation stays manual. Phased rollout.
Bank holiday — no transfers today Automation Config FinOps T1 turns off Progression auto for the day in gear panel. Prep and approval can happen, transfers will not auto-execute. Payouts can be approved and manually held until next day.
New FinOps T2 joins the team Permission T2 role has no hold/automation/manual buttons visible. Read-only view of holds and payouts. No config needed.

Current implementation

  1. Hold banners: ActiveHoldBanners groups Approval + Begin Transfer holds into single "Progression" row. Shows logical hold count. Collapsible detail with level badges (Fleet/Merchant/Payout).
  2. Preparation hold: Blocks "Prepare Payout" dialog. Manual prep cannot proceed. Auto prep is blocked.
  3. Progression hold: Blocks approval, transfer, and resubmit. UI shows hold reason. Manual progression and resubmit are disabled.
  4. Payout preview: "Prepare Payout" dialog includes mandatory settlement date, payout breakdown preview, and "Create" button.
  5. Automation Config: Gear icon panel shows Preparation toggle and Progression toggle (fleet + merchant level). Hold warnings displayed. "Holds always override automation" note visible.
